Birthday Paradox Simulation

This Java-based simulation explores the mathematical Birthday Paradox, which calculates the probability of shared birthdays within varying group sizes.

The application runs experimental trials for groups of 5 to 100 people, utilizing a random generator to simulate birthdays and calculate shared-date percentages over 20 trials per group.

Following clean code and TDD principles, I implemented modular classes for Simulation, Experiment, and Trial to maintain a clear separation of concerns and ensure mathematical accuracy.

Elevator System Simulator

Final implementation UML diagram showing class inheritance and relationships. (Click to enlarge)

Project Overview

This project is a discrete round elevator simulation designed to model complex building logistics through the lens of Object Oriented Programming. The system functions as a high level engine processing passenger requests across four distinct elevator models: Standard, Express, Freight, and Penthouse. Each unit operates under unique movement and access constraints, requiring a robust architecture to manage real time state transitions and floor restricted policies.

Functional Requirements and Logic

The simulation operates in discrete rounds to mimic realistic mechanical delays. Elevators move a maximum of one floor per round and must remain idle when picking up or dropping off passengers. Furthermore, if a unit needs to change direction, it incorporates an additional idle round to simulate the physical transition. The system follows a dynamic logic loop that remains active until every passenger call has been serviced and all elevator units have successfully returned to the ground floor lobby.

Technical Design and SOLID Principles

I designed the system architecture around an abstract Elevator class to define shared state while allowing specific subclasses to override behavior for unique policies. This design strictly adheres to SOLID principles, particularly the Open-Closed Principle, by enabling the seamless addition of new elevator types without modifying the core simulation engine. My design decisions focused on the four pillars of OOP, specifically using polymorphism to allow the scheduler to interact with any elevator type through a unified interface.

Test-Driven Development (TDD) and Implementation Role

A central focus of my work on this project was the rigorous application of Test-Driven Development (TDD). I prioritized a "test-first" workflow, authoring comprehensive JUnit test suites before implementing functional code. This process ensured that every component met strict requirements for reliability and edge case handling, such as simultaneous floor requests and capacity limits.

Professionally, I was responsible for implementing the Simulator engine, which serves as the central discrete-time controller, and the Scheduler architecture, including the core interface, the SimpleScheduler, and the ComplexScheduler. Additionally, I led the testing phase for these components, authoring rigorous test suites to verify mathematical accuracy in the proximity-based assignment logic.

Engaging deeply with TDD taught me the value of planning and verifying the reliability of scheduling algorithms before full integration. I gained practical experience in managing complex state transitions and utilizing GitHub for effective team collaboration and code reviews.
